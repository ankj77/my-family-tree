# Family Tree Digitization — Design

**Date:** 2026-07-12
**Status:** Approved

## Goal

Digitize a physical, multi-generation family tree (a folded paper chart, photographed)
into a durable, easy-to-read data file, and generate an interactive visual tree from it.

Two deliverables, one derived from the other:

1. A **YAML data file** that is the single source of truth — faithful, human-readable,
   easy to hand-edit and add to over time.
2. A **self-contained interactive HTML** family tree, generated from the YAML.

The source chart is patrilineal (father → children, arrows point down). Recent
generations are typed in English; the oldest ancestors are handwritten in Devanagari
(Hindi) with a few dates (e.g. *Sevak Ram, 1884 / Samvat 1941*, "Till 1945",
"Till 1925"). Several spots are taped-over or smudged and cannot be read confidently.

## Decisions (locked)

- **Data format:** YAML, source of truth, hand-editable.
- **Names:** stored in **both** English (`name`) and Devanagari (`name_hi`) where known.
- **Relationship shape:** flat list, each person points to their `father` (id reference).
- **Visual output:** single self-contained interactive **HTML** (pan/zoom/collapse/search),
  generated by a build step. Works offline (D3 and data inlined; no CDN).
- **Root:** exactly one — Sevak Ram (`sevakram`); everyone descends from him.
- **Uncertainty:** best-effort transcription now; unreadable/unknown entries flagged in the
  file and resolved later by editing + rebuilding.

## Data model

Top-level YAML list of people. One entry per person.

```yaml
- id: sevakram              # unique slug, required
  name: Sevak Ram           # English; required if name_hi absent
  name_hi: सेवकराम           # Devanagari; required if name absent
  born: "1884 (Samvat 1941)"  # optional, free text
  note: "root ancestor — all descend from him"  # optional, free text

- id: gaurishankar
  name: Dada Gaurishankar
  name_hi: दादा गौरीशंकर
  father: kannu             # id of parent; omitted ONLY for the root (sevakram)

- id: some_uncertain_name
  name: "?"
  father: kamal
  status: uncertain         # name can't be read; renders distinctly; must be resolved
  note: "taped over, below Kamal"

- id: ramjilal
  name: Ramjilal
  name_hi: रामजीलाल
  status: needs-parent      # name known, father unknown; to discuss + link
  note: "in Kannu's line, exact father unclear"
```

### Fields

| Field     | Required | Meaning |
|-----------|----------|---------|
| `id`      | yes      | Unique slug (lowercase, `_`-separated). Stable identity. |
| `name`    | one of `name`/`name_hi` | English name / transliteration, as written or best-effort. |
| `name_hi` | one of `name`/`name_hi` | Devanagari name, as written. |
| `father`  | no       | `id` of parent. Omitted **only** for the single root `sevakram`. |
| `born`    | no       | Free-text birth/date info (e.g. `"1884 (Samvat 1941)"`). |
| `note`    | no       | Free-text: location on chart, ambiguity, "Till 1945", etc. |
| `status`  | no       | `uncertain` (name unreadable) or `needs-parent` (father unknown). |

Notes:
- The "Till 1945" / "Till 1925" annotations are captured as `note` (not assumed to be
  death years) so nothing is lost while their meaning is ambiguous. They can be promoted
  to a `died` field later if confirmed.
- `uncertain` and `needs-parent` are distinct so the to-do list is precise.

## Build step (`build.py`, Python 3)

1. Read `family-tree.yaml` (PyYAML — the one dependency).
2. Validate (see below).
3. Assemble the parent → children tree from `father` pointers.
4. Emit a single self-contained `family-tree.html` with the tree data inlined as JSON and
   D3 (v7) vendored inline — no network needed to view.
5. Print a summary.

Run: `python3 build.py`

## Interactive HTML viewer

- Vertical, top-down **collapsible** tree (D3), rooted at Sevak Ram.
- **Pan + zoom** (drag / scroll / pinch).
- **Click a person → highlight the full lineage** up to the root.
- **Collapse/expand** any branch to tame the size.
- **Search box** to jump to and center a name (matches English or Devanagari).
- Each node shows **English name with Devanagari below it**; a language toggle
  (EN / हिंदी / both) switches the display.
- **`uncertain`** nodes: dashed border + `?` badge; hover shows the note.
- **`needs-parent`** people: parked in a visible "Unlinked — to place" area, not lost.
- `born`/`note` shown as subtitle or on hover.

## Validation & error handling

`build.py` **fails loudly** (non-zero exit, clear message) on:
- duplicate `id`
- a `father` referencing a non-existent `id`
- a cycle in the `father` chain
- a person with neither `name` nor `name_hi`
- more than one root (a non-`needs-parent` person other than `sevakram` with no `father`)

`build.py` **warns but succeeds** on:
- `status: needs-parent` people (lists them)

On success, prints a summary: total people, generations (max depth), and counts of
`uncertain` and `needs-parent` — the running to-do list.

## Testing (TDD)

Unit tests for the validator/assembler:
- duplicate id → error
- missing father → error
- cycle → error
- person with no name at all → error
- two roots → error
- valid fixture → correct assembled tree (parent/children, depth)
- `needs-parent` fixture → warning, still builds
- generated HTML contains every person's name (English and Devanagari)

Small YAML fixtures under `tests/`.

## Transcription workflow

During implementation, produce the initial `family-tree.yaml` from the photograph:
best-effort for every name, with:
- unreadable names → `status: uncertain` + a `note` locating them on the chart,
- known names whose father is unclear → `status: needs-parent` + a `note`.

The build summary's `uncertain` / `needs-parent` counts are the to-do list. Resolve by
editing the YAML and re-running `build.py`; sit down together to link the `needs-parent`
people to the correct fathers.

## Out of scope (YAGNI)

- Spouses / marriages / maternal lines (chart is patrilineal only).
- GEDCOM or genealogy-app export (can be added later from the YAML if wanted).
- Editing through a UI (edits are done in the YAML file).
- Photos, biographies, or other per-person media.
